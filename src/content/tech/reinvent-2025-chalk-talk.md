## はじめに

現在（12/1）ラスベガスにて、世界最大規模の AWS イベントである、re:Invent が開催されています。

https://reinvent.awsevents.com/

ありがたいことに今回参加させていただいており、スケールの大きさに驚かされる日々を送っています。

そんな中、せっかくなら「英語を浴びて何もわからなかった...」という敗北経験をしたいという気持ちが湧いてきたので、当初は予定していなかった Chalk talk に参加することにしました。その際の感想などをメモベースで共有したいと思います。

### タイトル

Build long running agents with Amazon Bedrock AgentCore

### 概要

上記ページの概要を Nani翻訳 にて和訳しました。

以下に記載の翻訳文の通り、Bedrock AgentCore Runtime と Memory を用いた長時間稼働するエージェント構築に関するセッションです。

>このチョークトークでは、複数日にわたる研究プロジェクトのような、長時間にわたる複雑なやり取りを実行できるAIエージェントの構築方法を探ります。
>
>Amazon Bedrock AgentCore Runtime & Memoryを使用して、タスクを一時停止および再開したり、数時間または数日にわたって状態を維持したり、複雑なワークフローを処理したりできるエージェントを構築する方法を紹介します。
>
>セマンティック、エピソード、プロシージャルといった異なる種類のメモリを効果的に活用し、長時間にわたるやり取り全体で状態を維持できるエージェントを構築する方法を学びます。
>
>これらのテクニックを、研究エージェントやサポートチケットの解決といったユースケースに適用する方法について説明します。

Nani翻訳 については以下をご覧ください。

https://nani.now/ja

## セッション内容について

前半30分弱で、Runtime や Memory のお話がされ、残りの30分は質疑応答の時間でした。

### 前半（サービス解説）

:::note warn
上記アプリによる文字起こし・翻訳と、Claude Sonnet 4.5 による要約、セッション中に撮影したスライド画像をもとに記載しています
:::

- 即座に応答のある、通常のチャットボットと異なる、長時間稼働するエージェントには以下のような課題がある
	- セッションの中断処理
	- コンテキストの維持

- 以上のような課題を解決するためのサービスとして、Amazon Bedrock AgentCore には以下のようなサービスがある
	- Runtime
	- Memory

- 事例紹介：金融調査エージェント
    - 概要
    	- 不正検出や規制違反の調査を行うエージェントシステム
    - エージェントの構成
    	- Supervisor
    	- Retrieval
    	- Compliance Analysis
    - ポイント
    	- 内部データベースからは即座に情報を取得
    	- 複数の金融機関からは3～5日で情報を取得
    		- Memory により、調査状況を保持
    		- S3 + Lambda でデータ取得後自動で動作

### 後半（質疑応答）

私が理解できた範囲の質疑応答のみ抜粋しています。

- Q. 不正検出を機械学習で実施しているが、これをどう組み込むか？
	- A. 機械学習モデルをツールとしてエージェントに提供
- Q. VPC内でのメモリ管理とセキュリティはどうなっているか？
	- A. GA後からVPCサポート対応、IAMポリシーを用いたアクセス制御が可能
- Q. エージェントの適切な分割方法
	- A. 1つのエージェントで済むなら1つ・エージェントが扱うデータの量を意識するべき

### セッション終了後に登壇者に質問してみた

全然英語を話せないのですが、聞きたいことはあったのでチャッピー君と一緒に考えて質問してみました。インシデント対応を実施（補助）する Ambient な Agent における記憶の考え方を聞いてみたかったのですが、割とよくある話をしていただき、その話を聞くまでにも結構時間を要してしまったので切り上げてしまいました。

これも思い付きで突撃してしまった形なので、せめて聞きたいことをちゃんと翻訳した状態で持っていけばよかったかなと反省しています。

しかし、「Take your time.（ゆっくりでいいよ）」と言ってくださるなどとても優しく対応していただけたので、他のセッションでも質問しようと思えたのはよかったと思います。
